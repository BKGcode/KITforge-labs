## 1. Brand DNA

### Tagline
**Purpose-built Unity tools for real workflows.**

### One-line positioning
Focused tools that remove the repeated tasks, awkward steps, and unnecessary effort that quietly slow Unity teams down.

### Philosophy (verbatim from studio)
> "Useful beats bigger."

### Core values

| Value | Meaning | How it shows up |
|-------|---------|-----------------|
| **Focus over breadth** | One tool = one job, clearly scoped | No mega-suites. Each product has a one-sentence promise. |
| **Honesty over hype** | Zero superlatives, zero inflated claims | Store pages describe what the tool does, not what it "revolutionizes". |
| **Practical over impressive** | Utility first, features second | We ship when it's useful, not when it's feature-complete. |
| **Respect for time** | Small improvements compound | Setup must feel light. Value visible in under 2 minutes. |
| **Empathy for the mundane** | Repetitive work deserves attention | We solve the boring problems no one else finds exciting enough to fix. |
| **Deliberate restraint** | Not building everything | Saying no to scope creep is a feature, not a limitation. |

### What we are NOT

- Not playful or casual (no humor, no emoji, no quirky mascot)
- Not corporate (no "leverage synergies", no "enterprise-grade solutions")
- Not aggressive (never name or attack competitors)
- Not salesy (no urgency tactics, no artificial scarcity, no "limited time")
- Not academic (no jargon-heavy explanations, speak to the human not the compiler)

## 2. Brand Voice

### Tone profile

| Axis | Position | Example |
|------|----------|---------|
| Formal ↔ Casual | Calm professional | "This tool finds missing references before they become build errors." |
| Confident ↔ Humble | Quietly confident | "It works. Here's how." — never "The BEST reference finder!" |
| Technical ↔ Accessible | Accessible with technical depth available | Store page = accessible. Documentation = technical. |
| Warm ↔ Cold | Warm but measured | Empathy without cheering. "We know this step is annoying." |
| Verbose ↔ Concise | Concise | Every sentence earns its spot. Cut the rest. |

### Writing rules

1. **Lead with the problem, not the product.** First sentence = what hurts. Second sentence = how this helps.
2. **Use plain verbs.** "Find", "clean", "check", "remove", "show". Not "leverage", "empower", "unlock".
3. **Avoid superlatives.** No "best", "most powerful", "revolutionary", "ultimate". Ever.
4. **Specifics over generics.** "Scans 500 prefabs in 2 seconds" beats "blazing fast performance".
5. **Acknowledge limits.** If the tool doesn't do something, say so. Honesty builds trust faster than silence.
6. **One promise per product.** If the store page needs two paragraphs to explain what it does, the scope is wrong.
7. **English for all public copy.** Store pages, README, documentation, social media — all in English.

### Vocabulary guide

| Use | Avoid |
|-----|-------|
| focused | powerful |
| practical | professional-grade |
| useful | essential |
| clear scope | comprehensive |
| day-to-day | mission-critical |
| reduces effort | saves hours |
| finds / shows / cleans | detects / eliminates / eradicates |
| works with | seamlessly integrates |
| lightweight | zero-overhead |

## 3. Customer Segments

Three segments, matched to the website positioning. KITforge does NOT target students or hobbyists — the brand speaks to people doing production work.

### Segment A — Studios

**Who:** Small-to-mid Unity studios (3–20 people) in active production.

**Trigger:** Process issues multiply across the team. One awkward step becomes 10 awkward steps across 10 people.

**What they buy:** Reliability, non-intrusiveness, source code access. They need tools that don't impose architecture, don't break their project, and can be debugged if something goes wrong.

**How they evaluate:**
- Does it compile clean in our project?
- Can we remove it without breaking anything?
- Is the source readable?
- Is there a public support channel with visible history?

**Price sensitivity:** Low-moderate. Will pay $40–60 for tools that save real time. ROI is obvious when multiplied by team size.

**Communication priority:** API reference, compatibility matrix, integration notes, changelog.

### Segment B — Solo Professionals

**Who:** Experienced solo developers or freelancers shipping real projects. Not hobbyists.

**Trigger:** Repeated tasks eat focus. They need to stay in flow state and can't afford to waste 20 minutes on setup every session.

**What they buy:** Immediate value, fast setup, clear documentation. They want to import, open, use, and forget it's a third-party tool.

**How they evaluate:**
- Does it work in under 2 minutes?
- Is the quick-start clear?
- Does it feel native to Unity?
- Is the price fair for one person?

**Price sensitivity:** Moderate. Will pay $15–35 for tools with clear daily value. Very sensitive to perceived "bloat" — if 70% of features don't apply, they feel overcharged.

**Communication priority:** Quick-start guide, GIF demos, clear feature list, honest limitations.

### Segment C — Prototype Teams

**Who:** Teams (or solo devs) in rapid prototyping phases. Game jams, proof-of-concept, pitch builds.

**Trigger:** Speed matters, but so does not drowning in technical debt from quick hacks.

**What they buy:** Speed of setup, no-config-required tools, reusable across projects.

**How they evaluate:**
- Can I use this across all my prototypes?
- Does it work immediately on import?
- Will it slow down my iteration loop?

**Price sensitivity:** High. Prototyping budgets are small. Freemium or low price points work best here.

**Communication priority:** Demo scene, one-click setup, visual result fast.

## 4. Asset Store Positioning

### How KITforge differentiates

The Asset Store is noisy. Most publishers compete on feature count and superlatives. KITforge differentiates by **tone and trust**.

| Competitor pattern | KITforge counter-position |
|---|---|
| "The most powerful X ever" | "X, clearly scoped and useful from day one." |
| Feature list with 40 bullet points | 5–8 features, each with a one-line explanation |
| Screenshots with everything visible | Clean screenshots focused on the one thing the tool does |
| "Works with everything!" | Explicit compatibility table with honest "not tested" entries |
| DLL / obfuscated code | Full C# source, always |
| Support only via private Discord | Public forum + email. Transparent issue tracking. |

### Positioning statement (internal, not published)

> KITforge labs makes focused Unity tools for professionals who are tired of bloated solutions and inflated promises. We sell one clearly scoped tool at a time, with honest copy, full source code, and documentation that respects the buyer's time.

### Category strategy

Start in **Tools > Utilities** and **Tools > Editor Extensions**. These categories have high traffic and reward clarity over complexity.

Shader products go to **Shaders > Fullscreen & Camera Effects** or **Shaders > Substances & Effects** depending on the pack.

## 5. AI Policy

### Position
KITforge labs uses AI as a development tool — primarily for testing, code review, and documentation iteration. Like any other tool in the pipeline.

### Unity compliance
Unity requires publishers to declare AI usage in asset creation. KITforge complies with this requirement accurately and completely on every submission.

### Communication approach
- **Do:** Declare AI usage where Unity policy requires it, honestly and without drama.
- **Do:** Treat it as normal. "We use modern development tools including AI-assisted testing."
- **Don't:** Make AI a selling point or marketing angle.
- **Don't:** Hide or minimize usage beyond what policy requires.
- **Don't:** Position the studio as "AI-powered" or "AI-built" — that invites the asset-flip stigma.

### Anti-stigma measures
The risk of being labeled "AI slop" exists for any new publisher in 2025–2026. Countermeasures:
1. **Full source code** — always. Readable, well-named, idiomatic C#. AI-generated code that looks bad would be caught in code review before shipping.
2. **Opinionated architecture** — tools follow documented conventions (KF_NamingConventions, unity-rules). Consistency signals human oversight.
3. **Quality of store page** — screenshots, descriptions, and demos show care and intention. No placeholder text, no generic stock imagery.
4. **Post-launch engagement** — responding to issues publicly and quickly signals a real team behind the product.

## 6. Anti-Patterns (things KITforge NEVER does)

### Communication
- Never compare directly to a named competitor in public copy
- Never use "best", "ultimate", "revolutionary", "game-changing"
- Never promise features that are planned but not shipped
- Never use fake urgency ("only X left!", "price going up!")
- Never use testimonials without permission
- Never delete or hide negative feedback

### Support
- Never require a 5-star review for support
- Never use private Discord as the only support channel
- Never dismiss a bug report without investigating
- Never go silent on a reported issue for more than 72 hours

### Product
- Never ship without a demo scene
- Never ship without source code
- Never ship without a README
- Never add unnecessary dependencies
- Never include dev/test files in the shipping package
- Never break the buyer's project when the tool is removed
